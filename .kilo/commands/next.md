读取 `TASKS.md` 和 `PROJECT.md`，输出下一步建议。

## 分析步骤

### 1. 扫描全部任务

读取 TASKS.md 中所有表格行（`| PX-XX |` 格式），按 Phase 分组，记录每条任务的：状态、优先级、涉及文件是否为空。

### 2. 状态一致性检查

对比表格行状态 vs 详情区块状态，发现不一致时标记。不一致时建议先修复再决策。

### 3. 跨任务依赖检查

识别未完成任务之间的依赖关系。被依赖的任务应排在前面。

### 4. 决策路由

```
TASKS.md 分析
│
├─ 有状态不一致 → 建议修复 TASKS.md
│
├─ 有未完成任务
│   ├─ P0 未开始 + 涉及文件空 → 建议调研
│   ├─ P0 未开始 + 涉及文件已填充 → 建议实现
│   ├─ P1/P2 未开始 → 按依赖排序，推荐无依赖的先推进
│   └─ 大任务可拆分 → 建议拆为子任务
│
└─ 全部完成 → 读 PROJECT.md，产品维度建议
    ├─ 自举路线当前 Level 分析
    │   当前是「Level 4 — 能吃」，问是否向更高级推进
    ├─ 远期规划中提到但未入 TASKS.md 的功能
    │   P36 dishcli 四协议多路复用
    │   P46 dishcli WebSocket 守护进程
    └─ 产品方向建议
        测试覆盖 / CI 完善 / 用户反馈闭环 / 性能优化
        新功能提案讨论
```

### 5. 全部完成时的建议思路

当所有任务完成时，参考以下维度输出建议（不要全部列出，挑最合适的 1-2 个）：

- **自举进阶**: 当前 Level 4（能吃），下一步可考虑 Level 5 方向（如自动化测试覆盖、CI 流水线）
- **远期规划落地**: TASKS.md 中提到 P36/P46 等技术方向，可转化为新 Phase
- **用户反馈驱动**: 收集真实使用中的痛点，转化为任务
- **技术债务**: 梳理注释规范、类型覆盖、错误处理边界等
- **新功能提案**: 基于 KCode 产品定位提出值得讨论的新功能方向

## 原则

- **不读 git status/git log/分支信息** — 只基于任务和产品维度做决策
- **不自动执行** — 输出建议 + 理由，等待用户确认
- **状态一致性优先** — 发现矛盾时先修复再推进
- **依赖优先** — 被依赖的任务排在前面
- **全部完成时做产品维度建议** — 基于 PROJECT.md 和已有能力推断下一步方向
