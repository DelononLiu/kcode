## 致命架构问题

### 1. 两套渲染路径互斥竞争 ⚠️⚠️⚠️

**路径 A (直连 DOM)**: `flowCards.ts` 的 `showGoalConfirmationCard`, `handleShowPlanProposal` 等函数直接向 `#chat-messages` append DOM 元素。

**路径 B (消息回放)**: `messageRenderer.ts` 的 `renderMessages()` 执行 `container.innerHTML = ''` 清空全部 DOM，然后从消息数组重建。

这两个路径产出**不同的卡片 HTML 结构**：

| 卡片 | 路径 A (flowCards.ts) | 路径 B (messageRenderer.ts addMessageElement) |
|------|----------------------|----------------------------------------------|
| 目标卡 | `showGoalConfirmationCard` — 含编辑模式、分类标签 | `goal_confirmation` handler — 简化版 |
| 计划卡 | `handleShowPlanProposal` — 含行内编辑 | `plan_proposal` handler — 简化版 |
| 执行卡 | `showExecuteConfirmation` — 独立函数(从未被调用) | `execute_confirmation` handler — 从 loadMessages 触发 |
| 自验卡 | `showSelfVerifyConfirmation` — 独立函数(从未被调用) | `self_verify_confirmation` handler — 从 loadMessages 触发 |

**冲突触发点**: 当路径 B 的 `loadMessages` (带 `render:true`) 到达时:
1. `container.innerHTML = ''` 销毁路径 A 创建的所有卡片
2. `renderMessages` 从消息数组重建 → 使用路径 B 的 `addMessageElement`
3. 两个路径产出的卡片 UI 不同 → 用户看到"卡片消失"或"样式突变"

**关键**: 路径 A 的 `showExecuteConfirmation` / `showSelfVerifyConfirmation` **从未被调用** (`app.ts` 注册了处理函数但没有发送方)。只有路径 B 的 `addMessageElement` 中的 handler 会渲染执行/自验卡片——但它们只有在 `loadMessages` 带 `render:true` 时才会被触发。

### 2. `removeGoalConfirmationCard()` 核弹 ⚠️⚠️

```typescript
// flowCards.ts:551
export function removeGoalConfirmationCard() {
    document.querySelectorAll('.msg-card').forEach(el => el.remove()); // 删除所有卡片
}
```

这个函数名为"移除目标确认卡片"，实际删除了页面上**所有** `.msg-card` 元素。当 `showGoalConfirmationCard` 被调用时（发生在需求阶段），它会先清空所有卡片再创建目标卡片。

### 3. 全局状态无结构依赖 ⚠️⚠️

`G` 对象 (`state.ts`, 39 个字段) 被所有 webview 模块直接读写:

- `G.activeTaskPhase` 在 `app.ts` 的 `loadMessages` / `updateTaskInfo` handler 中设置
- 同时被 `messageRenderer.ts` 的 `renderMessages` / `addMessageElement` 读取
- 同时被 `chatStream.ts` 的 `appendToChatMessages` 读取
- 同时被 `taskView.ts` 的 `foldPhase` / `updateHeaderRow2` 读取

任何中间件的 `postMessage` 处理都可能改变 `G.activeTaskPhase`，破坏其他正在进行的渲染。

**重复状态**: `_tabGroup`, `_mergeState`, `activeToolCallElements`, `reviewChangesMap`, `selectedReviewFileIdx`, `_demoCards` 在 `G` 对象上声明了但实际使用模块级变量——两套状态各自存在。

### 4. loadMessages 渲染触发不可靠 ⚠️

```typescript
// app.ts:317
if (isNewTask || message.render || message.taskStatus === 'in_review') {
    renderMessages(message.messages || []);
}
```

渲染触发条件依赖三个 flag，其中 `message.render` 是一个**非标准属性**，只在 `TaskStreamHandler.onDone` 的特定分支（execute/self_verify 确认消息）中设为 `true`。其余 16+ 个 `loadMessages` 调用点都**不带** `render: true`，导致消息更新了但 UI 不刷新。

### 5. 卡片生命周期无统一管理 ⚠️

四个卡片（goal / plan / execute / self_verify）的创建、交互、确认/驳回、回退/重试没有一个统一的"卡片状态机"。每种卡片在路径 A 和路径 B 中各有一套写死的逻辑:

```
目标卡片: 路径A(showGoalConfirmationCard) + 路径B(goal_confirmation handler)
计划卡片: 路径A(handleShowPlanProposal)   + 路径B(plan_proposal handler)
执行卡片: 路径A(showExecuteConfirmation)  + 路径B(execute_confirmation handler)
自验卡片: 路径A(showSelfVerifyConfirmation) + 路径B(self_verify_confirmation handler)
```

每个路径的 button 交互逻辑独立编写，修改时需要同步两处。任何一边不同步就是 bug。

### 6. 协议解析与 UI 渲染时序耦合 ⚠️

`TaskStreamHandler.onDone()` 是问题的集中爆发点:

```
onDone 被调用
  → 遍历 activeToolCalls, 处理 tool output 中的 TASK_UPDATE (可能触发状态变更)
  → 检查 genResult.planProposed / executeFinished / selfVerifyFinished
  → 根据检查结果走不同分支
  → 每个分支里: storeMessage + sendTaskInfo + sendNodePanelUpdate + loadMessages
  → 最后 resetGeneration 清空所有标记
```

`genResult` 的标志（`planProposed`, `executeFinished`）在同一轮 AI 响应中可能**同时为 true**（如果 AI 在 execute 阶段先完成了 plan_step_update 触发了 `propose_plan`，然后又发了 `finish_execute`）。由于 `if-else if` 链条的短路特性，后一个标志可能被第一个匹配的分支吞掉。

### 7. 扩展侧 View→WebView 协议碎片化

从 `Panel.ts` 的消息路由注册（`router.on`）来看，当前有 **30+ 种不同的消息类型** 在扩展和 WebView 之间传递。部分消息类型处理了就直接操作 DOM（路径 A），部分通过 `loadMessages` 触发重渲染（路径 B），没有一个统一的"UI 更新协议"。

### 8. foldPhase 动态 DOM 重排

`foldPhase` (`taskView.ts:44`) 在 `renderMessages` 末尾执行，将所有已完成阶段的 `.tv4-phase-group` 子元素提取、分类、重组到 `tv4-pg-toggle` + `tv4-pg-body` 结构中。这发生在 DOM 已经完全渲染之后，是二次重排。如果有任何异步操作在 foldPhase 之后追加元素到被折叠的 group，这些元素就会"消失"。

### 安全重写策略: 新文件, 不动旧代码

**核心原则**: 旧代码一行不改。所有新代码写在独立目录下，通过开关切换。

#### 文件组织

```
src/view/webview/taskv2/               ← 新增目录
  ├── state.ts                          StateManager + 类型定义
  ├── basePipeline.ts                   共用渲染管线 (stream/markdown/toolCards/messages)
  ├── renderManager.ts                  统一渲染入口
  ├── viewStrategy.ts                   ViewStrategy 接口
  ├── taskStrategy.ts                   任务模式策略实现
  ├── assistantStrategy.ts              小助手模式策略实现
  ├── cardRenderer.ts                   统一卡片渲染 (5 种卡片)
  └── streamManager.ts                 流式内容管理

src/view/TaskViewBridgeV2.ts           ← 新增: 扩展↔webview 桥接

kilo.json:
  "taskViewV2": false                   ← 开关 (开发期 false, 验证通过后 true)
```

**关键约束**: 重构期间不对这 11 个旧文件做任何改动: `flowCards.ts`, `messageRenderer.ts`, `chatStream.ts`, `app.ts`, `taskView.ts`, `TaskStreamHandler.ts`, `TaskFlowHandler.ts`, `state.ts`, `cardBuilder.ts`, `Panel.ts`, `chatPanelHtml.ts`。

#### 开关机制

```
kilo.json
  └─ "taskViewV2": false
       │
       ├─ false → Panel.ts 走旧路径 (TaskFlowHandler + app.ts + flowCards + ...)
       │          小助手模式完全不受影响 (始终走旧路径)
       │
       └─ true  → Panel.ts 走新路径 (TaskViewBridgeV2 → stateManager → renderManager → taskStrategy)
                  小助手模式继续走旧路径 (直到 Phase 3 才迁移)
```

Panel.ts 中的切换逻辑 (示意):

```typescript
// Panel.ts constructor 或 loadTask 中
if (this.configService.get<boolean>('taskViewV2', false)) {
    this.taskViewBridge = new TaskViewBridgeV2(this.ctx);
} else {
    // 现有代码完全不变
    this.flowHandler = new TaskFlowHandler(ctx);
    this.sessionHandler = new TaskSessionHandler(ctx);
    // ... 现有 router.on 注册全部保留
}
```

#### 并行加载流程

```
用户点击 task → loadTask(taskId)
  │
  ├─ taskViewV2 = false (旧)
  │    Panel.loadTask() → TaskFlowHandler.sendTaskMessages() → loadMessages
  │    → app.ts 消息路由 → flowCards 直连 DOM + messageRenderer 回放渲染
  │    → 现有行为完全一致
  │
  └─ taskViewV2 = true (新)
       TaskViewBridgeV2.init(taskId) → send('messages-sync', messages)
       → renderManager.render() → basePipeline.renderMessageList()
       → taskStrategy.renderHeader()
       → 新架构行为
```

```
│                      ▼                                    │
│              renderManager.ts         ◀── 新增            │
│            (统一入口, 只此一处写 DOM)                      │
│              /            \                               │
│             ▼              ▼                              │
│     basePipeline            viewStrategy  ← 可插拔       │
│  ┌──────────────────┐  ┌─────────────────────────┐       │
│  │ • streamEngine   │  │ TaskStrategy             │       │
│  │ • markdownEngine │  │   (5 阶段卡片/确认/驳回)   │       │
│  │ • toolCardEngine │  ├─────────────────────────┤       │
│  │ • messageEngine  │  │ AssistantStrategy        │       │
│  └──────────────────┘  │   (自由对话/引导系统)      │       │
│          │             ├─────────────────────────┤       │
│          │             │ AssistantV2Strategy      │ ← 未来 │
│          │             │   (Code Review 助手等)    │       │
│          │             └─────────────────────────┘       │
│          └─────────────┬──────────┘                       │
│                        ▼                                  │
│                 #chat-messages (DOM)                      │
└───────────────────────────────────────────────────────────┘
```

**核心原则**:

1. **单一渲染路径** — 不再有"直连 DOM"和"消息回放"两套。所有 UI 变更通过 state → render flow。

2. **单向数据流** — 扩展 → state delta → 订阅者 → DOM。不存在 WebView 内部直接 DOM 操作。

3. **Base + Strategy 分离** — `basePipeline` 负责所有模式共有的渲染能力（流式追加、markdown、工具卡片、消息列表）。`viewStrategy` 负责模式特有的 UI 和行为（任务卡片、引导对话等）。新增模式只需实现 `ViewStrategy` 接口。

4. **卡片统一生命周期** — 一个 `TaskCard` 抽象, 支持 `proposed → confirmed → completed` 状态。按钮、编辑模式、回退全部统一定义。

5. **流式 + 批量分离** — 流式输出走 streamEngine (增量 append 到 DOM 尾部)。阶段卡片渲染只在 `[stream-done]` 事件后批量执行。

6. **G 对象消除** — 用 `stateManager` 替换。所有模块只能通过 `subscribe` / `dispatch` 访问状态。

### 视图策略模式 (View Strategy Pattern)

核心设计是 **组合优于继承**。`basePipeline` 和 `viewStrategy` 是两个正交的维度：

| 维度 | 职责 | 变更频率 | 谁负责 |
|------|------|---------|--------|
| basePipeline | 流式文本追加、markdown 渲染、工具卡片合并/折叠、消息时间线、滚动控制 | 低 | 基础层 |
| viewStrategy | 阶段卡片生命周期、确认/驳回交互、header 布局、引导流程、view 切换 | 高 | 策略层 |

#### ViewStrategy 接口

```typescript
interface ViewStrategy {
    /** 渲染该模式特有的 header/布局 */
    renderHeader(state: AppState): void;

    /** 流式输出完成后的处理（如任务模式检查 genResult 决定是否显示卡片） */
    onStreamDone(state: AppState, result: StreamResult): void;

    /** 用户交互（确认/驳回/修改等） */
    onUserAction(state: AppState, action: UserAction): void;

    /** 该模式是否需要阶段面板 */
    showPhasePanel(): boolean;

    /** 该模式是否需要折叠历史阶段 */
    shouldFoldPhases(): boolean;

    /** 该模式是否需要右侧产出物面板 */
    showOutputPanel(): boolean;
}
```

#### 现有两种策略

```typescript
// 任务模式
const taskStrategy: ViewStrategy = {
    renderHeader: (s) => renderTaskHeader(s.taskInfo),
    onStreamDone: (s, r) => {
        if (r.executeFinished) showExecuteCard(s.activeTaskId);
        else if (r.planProposed) showPlanCard(s.activeTaskId);
        // ...
    },
    onUserAction: (s, a) => {
        switch (a.type) {
            case 'confirmGoal': dispatchConfirmGoal(s, a);
            case 'approveReview': dispatchApproveReview(s, a);
            // ...
        }
    },
    showPhasePanel: () => true,
    shouldFoldPhases: () => true,
    showOutputPanel: () => true,
};

// 小助手模式
const assistantStrategy: ViewStrategy = {
    renderHeader: (s) => renderAssistantHeader(),
    onStreamDone: (s, r) => { /* 只做消息持久化, 不显示卡片 */ },
    onUserAction: (s, a) => {
        if (a.type === 'convertToTask') dispatchConvertToTask(s, a);
        // 小助手模式下大部分 action 直接走 basePipeline
    },
    showPhasePanel: () => false,
    shouldFoldPhases: () => false,
    showOutputPanel: () => false,
};
```

#### 新增第三个模式

假设未来要加一个「Code Review 专用助手」——不闲聊，只分析 PR diff，自动生成 review 意见：

```typescript
const reviewStrategy: ViewStrategy = {
    renderHeader: (s) => renderReviewHeader(s.prUrl, s.branchName),
    onStreamDone: (s, r) => {
        // 自动提取 diff summary, 显示 review 结果卡片
        showReviewResultCard(s.activeTaskId, r.cleanedText);
    },
    onUserAction: (s, a) => {
        // 只接受 approve / reject / comment
        if (a.type === 'approveDiff') dispatchApproveDiff(s, a);
        if (a.type === 'rejectLine') dispatchRejectLine(s, a);
    },
    showPhasePanel: () => false,
    shouldFoldPhases: () => false,
    showOutputPanel: () => true,  // 显示 diff panel
};

renderManager.registerStrategy('reviewAssistant', reviewStrategy);
```

不需要修改 basePipeline，不需要新增 PostMessage type，不需要动 TaskFlow。30 行接口实现即可。

#### basePipeline 负责的共用能力

```typescript
interface BasePipeline {
    /** 流式文本增量追加, 自动处理 markdown 渲染节流 */
    appendStreamChunk(chunk: string): void;

    /** 流式结束, 刷新缓冲区, 移除游标动画 */
    finalizeStream(): void;

    /** 从消息数组全量重建 DOM (仅 task 切换时) */
    renderMessageList(messages: ChatMessage[]): void;

    /** 追加单条消息 (用户消息、tool call、系统消息) */
    appendMessage(msg: ChatMessage): void;

    /** 追加/更新工具调用卡片 (bash/read/write/grep 等) */
    upsertToolCard(toolCall: ToolCallState): void;

    /** 工具调用合并 + 思考折叠 (thinking → tools 时序线) */
    mergeToolTimeline(): void;
}
```

这些能力在所有 viewMode 下完全一致——任务模式的 `npm install` 输出和小助手模式的 `npm install` 输出渲染完全一样。

## 新架构对小助手模式的影响

### 当前小助手架构（简要）

```
AssistantHandler (extension)
  ├ handleMessage() → sendPrompt → AssistantStreamHandler
  ├ showLanding() → PostMessage(updateNodePanel, updateTaskInfo)
  ├ startGuide() → 3 步引导对话
  ├ convertToTask() → 聊天转正式任务
  └ loadMessages() → PostMessage(loadMessages, taskType='assistant')

AssistantStreamHandler (extension, 55 行)
  ├ onText() → 追加 buffer → agentStreamUpdate
  └ onDone() → store messages → loadMessagesFn()

WebView: app.ts loadMessages handler (line 277-283)
  if (taskType === 'assistant'):
    showAssistantView()
    renderMessages(messages)   ← 直接调用, 无 render guard
    break                       ← 跳过所有 task 模式逻辑
```

### 关键发现：小助手模式已经不依赖 render guard

`loadMessages` handler 对 assistant 类型做了**早期返回**:

```typescript
if (message.taskType === 'assistant') {
    G.activeTaskId = message.taskId;
    G.activeTaskStatus = message.taskStatus || '';
    G.activeTaskType = message.taskType || '';
    showAssistantView();
    renderMessages(message.messages || []);  // ← 无条件渲染
    break;  // ← 直接返回, 跳过所有 task 模式逻辑
}
```

这意味着:
- 小助手模式没有 `isNewTask || render || ...` 的 guard
- 小助手模式没有 `foldPhase` 折叠逻辑
- 小助手模式没有卡片渲染（goal/plan/execute/self_verify cards）
- 小助手模式没有 phase tracking（`G.activeTaskPhase` 不用）
- 小助手模式的消息始终会被渲染

**结论**: 小助手模式已经接近新架构的"单一渲染路径"设计。它不需要修复。

### 新架构下 —— viewMode 决定 viewStrategy

小助手模式被视为 `viewStrategy` 的第二个实现。架构上两者是对等的：

```
stateManager.viewMode
  ├── 'task'      → renderManager.setStrategy(taskStrategy)
  └── 'assistant' → renderManager.setStrategy(assistantStrategy)
```

| 维度 | 当前小助手 | 新架构 (assistantStrategy) |
|------|-----------|---------------------------|
| 消息协议 | 沿用 ~10 种 task 消息类型 | 共用 3 种: `stream-chunk`, `stream-done`, `messages-sync` |
| 状态管理 | `G` 对象 + AssistantHandler 内部状态 | `stateManager` + `assistantState` 子状态 |
| view 切换 | `showAssistantView()` CSS toggle | 声明式: `stateManager.viewMode` 变更 → renderManager 自动切换 strategy |
| 流式渲染 | `agentStreamUpdate` → 直接追加 DOM | `stream-chunk` → basePipeline.streamEngine 追加 |
| 消息重载 | `loadMessages` → `renderMessages` | `messages-sync` → basePipeline.messageEngine 全量渲染 |
| 引导系统 | `_guideStep` + `_guideMessages[]` (硬编码) | assistantStrategy 内部状态机, 可配置 |
| 聊天转任务 | 直接操作 store + loadTask + refreshSidebar | dispatch `convertToTask` → TaskViewBridge 统一处理 |
| 常见渲染 | markdown + tool cards 和任务模式不同代码路径 | **同一套 basePipeline**, 零重复代码 |
| 扩展性 | 新助手需复制全部逻辑 | 新实现一个 ViewStrategy 接口即可 |

### 实际收益（对小助手）

1. **渲染代码零重复** — 小助手的 markdown/工具卡片/流式输出和任务模式完全共用 `basePipeline`。当前这些能力在两套代码里独立实现。

2. **viewMode 声明式切换** — `stateManager.viewMode = 'assistant'` 代替 jQuery 式 CSS toggle。renderManager 自动加载对应的 viewStrategy。

3. **添加新助手模式只需 30 行** — 实现 `ViewStrategy` 接口，`renderManager.registerStrategy('reviewAssistant', reviewStrategy)` 即可。

### 后期小助手重构可行性

**完全兼容**, 而且简化。小助手模式作为 strategy 实现，和任务模式共享 basePipeline 但独立维护自己的 UI 逻辑:

1. **代码量更小** — 当前 AssistantHandler 406 行 + AssistantStreamHandler 55 行 → 新结构下 assistantStrategy ~200 行（去掉流式渲染和 markdown, 交给 basePipeline）
2. **不再需要独立的 AssistantStreamHandler** — 流式处理由 `stream-chunk` → basePipeline.streamEngine 统一接管
3. **引导系统可独立迭代** — assistantStrategy 内部状态机不依赖 TaskFlow, 可以随时改引导步骤

### 重构顺序建议

```
Phase 1: basePipeline + taskStrategy (任务模式新架构)
    ↓ 验证通过
Phase 2: 裁剪旧代码, 保留小助手 (旧小助手暂时不动)
    ↓ 新任务模式稳定运行
Phase 3: 实现 assistantStrategy (复用 basePipeline)
    ↓
Phase 4+: 未来新增策略 (reviewAssistant, debugAssistant...) — 每次只需实现 ViewStrategy 接口
```
