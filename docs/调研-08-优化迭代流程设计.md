# KCode AI 自主优化迭代流程设计

> 不同任务类型需要不同的 flow 表达方式。现有系统已通过 `flowOverride` 支持跳过阶段，本设计在此基础上增加**阶段间迭代循环**能力，让 `biz_logic` 等适合迭代优化的任务类型拥有 `execute ↔ self_verify` 自动循环。

---

## 一、背景与问题

### 当前 flow 机制

每种 `TaskTemplate` 通过 `flowOverride` 定义自己的阶段顺序（未定义时使用默认值）：

```
默认 6 阶段:                           demand → goal → plan → execute → self_verify → review
代码评审 (code_review):               demand → goal → review
文档开发 (doc_dev):                    demand → goal → execute → review
```

`flowOverride` 仅支持**跳过阶段**，不支持**阶段间循环**。

### 问题

1. `biz_logic`（业务逻辑迭代）等需要多轮优化的任务类型，无法表达 execute ↔ self_verify 之间的自动循环
2. 自验阶段校验标准笼统，无分层校验能力
3. 无基线管理，改坏了只能靠 git 手动恢复
4. 缺少迭代决策机制（达标/超限/停滞的判断）

### 设计目标

- 不改动现有 6 阶段基础结构
- 在 `flowOverride` 同级新增 `flowIteration` 表达迭代能力
- 自验阶段实现**分层校验 + 三路决策出口**
- 正确性失败时自动回滚并重试
- 迭代完成后更新基线

---

## 二、核心概念：Flow 是 TaskTemplate 的属性

不同任务类型有不同的 flow：有的线性一次过（需求开发）、有的只审不改（代码评审）、有的需要迭代循环（业务逻辑迭代）。

每种 `TaskTemplate` 通过两个字段定义自己的 flow 行为：

| 字段 | 作用 | 现有示例 |
|------|------|---------|
| `flowOverride` | 定义阶段顺序，可跳过 | `doc_dev`: 跳 plan 和 self_verify |
| **`flowIteration`（新增）** | 定义哪些阶段间可循环 | `biz_logic`: execute ↔ self_verify |

这两个字段加在一起，完整表达一个任务类型的 flow 结构：

```
需求开发（新增功能）:
  flowOverride: 无（默认 6 阶段）
  flowIteration: 无
  → 线性一次过

业务逻辑迭代:
  flowOverride: 无（默认 6 阶段）
  flowIteration: { loopPhases: ['execute', 'self_verify'] }
  → execute ↔ self_verify 自动循环

代码评审:
  flowOverride: ['demand', 'goal', 'review']
  flowIteration: 无
  → 只审不改
```

### 在 Task 上的运行时表达

`TaskTemplate` 的预设编译到 `Task` 上，运行时由 flow 驱动：

```typescript
interface Task {
    // ...现有字段...
    flowOverride?: Array<Phase>;                          // 从 template 继承
    flowIteration?: {                                     // 从 template 继承 + goal 阶段配置
        enabled: boolean;
        loopPhases: ['execute', 'self_verify'];
        config: {
            correctnessTests: string[];                   // goal 阶段固化
            targets: Record<string, number>;              // goal 阶段固化
            iterationLimit: number;                       // goal 阶段固化
        };
        state: {                                          // 运行时状态
            currentIteration: number;
            stagnatedCount: number;
            baselines: Record<string, number>;
            history: IterationRecord[];
        };
    };
}
```

## 四、状态机变更

### 当前状态机

```
confirmExecuteDone()     → phase = self_verify
confirmSelfVerifyDone()  → phase = review
rejectReview()           → phase = execute（用户介入）
```

### 改造

只改 `parseTaskUpdate` 中 `finish_verify` 的处理分支：

```typescript
if (task.phase === 'self_verify' && payload.ACTION === 'finish_verify') {
    text = text.replace(match[0], '');
    this.accumulatedText.set(taskId, text);
    regex.lastIndex = 0;

    if (payload.DECISION === 'continue' && task.flowIteration?.enabled) {
        // 迭代循环：self_verify → execute，直接切 phase
        this.store.updateTaskPhase(taskId, 'execute');
        this.store.updateTaskStatus(taskId, 'active');
        this.store.addTimelineEntry(taskId, {
            timestamp: Date.now(),
            type: 'phase_change',
            summary: '自验通过 → 继续执行（迭代）',
        });
        this.delegate.onPhaseChanged(taskId);
    } else {
        // success / timeout / stagnation → review
        this.selfVerifyFinished.set(taskId, true);
        this.delegate.onSelfVerifyFinished(taskId);
    }
    continue;
}
```

不走 `confirmSelfVerifyDone()` 再 revert，因为迭代回切是 AI 协议层内部决策，不需要经过 UI 事件桥接。

### Protocol 扩展

`finish_verify` 协议增加决策字段：

```
[TASK_UPDATE]
ACTION: finish_verify
DECISION: continue          # continue | success | timeout | stagnation
METRICS: {"latency": 320}
ITERATION: 2
[/TASK_UPDATE]
```

`PROTOCOL_KEYS` 增加 `DECISION`、`METRICS`、`ITERATION`。

## 五、自验阶段分层校验

### 流程图

```
self_verify 进入
    │
    ├── Layer 1：正确性校验 ──────────────────────────────┐
    │                                                     │
    │   AI 执行 config.correctnessTests                   │
    │    ├── 全部通过 → Layer 2                            │
    │    └── 有失败 ── 自动修复重试 ──┐                    │
    │                      失败≤3次 ─┤                     │
    │                      失败>3次 ─┴─→ DECISION=continue  │
    │                                                     │
    ├── Layer 2：优化指标校验 ─────────────────────────────┤
    │                                                     │
    │   AI 测量当前指标，对比 baselines vs targets          │
    │   记录到 history                                      │
    │                                                     │
    ├── Layer 3：迭代决策 ────────────────────────────────┤
    │                                                     │
    │  条件                              → DECISION        │
    │  ───────────────────────           ─────────        │
    │  所有指标达标                       success          │
    │  未达标 + 未达上限 + 有改进          continue         │
    │  已达 iterationLimit                timeout          │
    │  连续 2 轮无改进                    stagnation       │
    │                                                     │
    └── 输出 finish_verify + DECISION ────────────────────┘
```

### Prompt 注入

在 `buildPhasePrompt` 中判断 `task.flowIteration?.enabled`，为 execute 和 self_verify 阶段追加迭代上下文。

self_verify 注入内容：

```
【迭代优化 - 第 N/limit 轮】
基线指标: latency=450ms, complexity=15
目标: latency≤200ms, complexity≤10
历史:
  iter 1: latency=380ms, complexity=13 [改进]
  iter 2: latency=350ms, complexity=12 [改进]

【正确性测试】
npm test
npm run lint

【决策规则】
1. 正确性测试失败 → 最多自动修复 3 次，仍失败则 DECISION=continue
2. 全部指标达标 → DECISION=success
3. 未达标 + 未达上限 + 有改进 → DECISION=continue
4. 已达上限 → DECISION=timeout
5. 连续 2 轮无改进 → DECISION=stagnation
```

execute 注入内容：

```
【迭代优化 - 第 N/limit 轮】
当前需要在前一轮基础上继续优化。
历史指标: ...
```

## 六、各任务类型的 Flow 定义

| 任务类型 | 子类型 | flowOverride | flowIteration | 形态 |
|---------|--------|-------------|---------------|------|
| **需求开发** | 新增功能 | 默认 6 阶段 | 无 | 标准线性 |
| | 业务逻辑迭代 | 默认 6 阶段 | `{loopPhases: [execute, self_verify]}` | **线性 + 迭代循环** |
| | 接口开发 | 默认 6 阶段 | 无 | 标准线性 |
| | 页面/组件 | 默认 6 阶段 | 无 | 标准线性 |
| | 文档开发 | `[demand, goal, execute, review]` | 无 | 跳 plan + self_verify |
| **问题分析** | 全部 | 默认 6 阶段 | 无 | 标准线性 |
| **代码评审** | 全部 | `[demand, goal, review]` | 无 | 只审不改 |
| **日志分析** | 全部 | `[demand, goal, review]` | 无 | 分析型 |
| **缺陷分析** | 全部 | 默认 6 阶段 | 无 | 标准线性（复杂缺陷可扩展） |

## 七、git worktree 隔离方案

在 execute 阶段使用 `git worktree` 隔离修改，避免污染主分支。

### 流程

```
Execute 进入
  ① git worktree add .kcode/worktree/iter-N baseline-branch
  ② 在 worktree 中修改代码、运行测试
  ③ finish_execute

Self-Verify（正确性失败 → DECISION=continue + rollback）
  ④ 删除 worktree: rm -rf .kcode/worktree/iter-N
  ⑤ git worktree prune
  ⑥ 从基线重建: git worktree add .kcode/worktree/iter-N baseline-branch
  ⑦ 回到 execute

Review（通过）
  ⑧ 合并 worktree 到主分支
  ⑨ 删除 worktree，标记新 baseline
```

### Task 中的 worktree 信息

```typescript
task.flowIteration.state.baselines = { /* 基线指标 */ };
// worktree 路径由 TaskFlow 在 execute 进入时计算并存储
```
