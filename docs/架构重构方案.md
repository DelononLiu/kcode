# KCode 整体架构重构方案

> 日期：2026-07-05
> 状态：提案
> 涉及：数据模型 / Agent 框架 / UI 渲染

---

## 设计原则

**小助手（Assistant）和任务（Task）是同一个对话模型的两种模式，不是两套系统。**

- 小助手 = `Task.type === 'chat'`，无阶段管理（没有 phase 流转），但对话机制、消息结构、折叠能力完全一致
- 任务模式 = `Task.type === 'task'`，在小助手能力之上增加阶段节点（goal → plan → execute → self_verify → review）
- 渲染层**不区分**任务和小助手——`MessageList` 组件同时服务于两种模式，只有 header/phase 部分不同
- 折叠功能适用于两种模式——AI 回答完成后中间输出自动折叠

---

## Context

KCode 目前有三个割裂的数据模型：

- **`types/index.ts`** — `Task`、`ChatMessage`（供 Store 持久化和 Extension 逻辑用）
- **`taskv3/types.ts`** — `AppState`、`Message`（供 WebView 渲染用）
- **`TaskFlow.ts`** — 手写状态机（Map 散列管理 phase 状态，`[TASK_UPDATE]` 协议解析）

三套不完全一致，手动同步，导致折叠、任务流频繁出 bug。

同时，`docs/Agent框架选型.md` 已论证 LangGraph.js 应作为新的 Local Agent 框架，但它目前仅是文档，从未落地。

**本方案的核心判断**：LangGraph.js 的 `StateGraph` + checkpoint 机制 和 UI 层需要的数据模型其实是**同一个问题**——即"Task 的状态应该由谁来管理、怎么流动"。必须统一设计，不能先改 UI 再改 LLM 层。

## 当前架构问题

### 问题 1: Task 数据结构分裂

```typescript
// types/index.ts — 存储用
interface ChatMessage {
    id: string; taskId: string;
    role: 'user' | 'agent' | 'tool';
    type?: 'text' | 'goal_confirmation' | 'tool_call' | ...;  // 13 种类型
    content: string;        // 要么纯文本，要么需要外部解析
    phase?: string;
    timestamp: number;
}

// taskv3/types.ts — 渲染用（不同步）
interface Message {
    id: string; taskId: string;
    role: 'user' | 'agent' | 'tool';
    type?: string;         // 字符串字面量，无约束
    content: string;        // 同样混合
    phase?: string;
    timestamp: number;
    streaming?: boolean;    // 渲染层字段
    collapsed?: boolean;    // 渲染层字段
    roundGroup?: string;    // 渲染层字段
    cardMeta?: {...};       // 控制逻辑伪装成数据
}
```

**结果**：`messages-sync` 时要把 `ChatMessage[]` 转成 `Message[]`，转换过程丢字段、加字段，两头不一致。

### 问题 2: 状态机手写 + 协议裸解析

`TaskFlow.ts` 的核心是一个 `Map<string, boolean>` 集群：

```typescript
private planProposed: Map<string, boolean> = new Map();
private goalProposed: Map<string, boolean> = new Map();
private executeFinished: Map<string, boolean> = new Map();
// ... 类似 Map 有 7-8 个
```

阶段迁移通过正则解析 AI 输出中的 `[TASK_UPDATE]...[/TASK_UPDATE]` 块：

```typescript
// TaskFlow.ts:194
text.replace(/\s*\[TASK_UPDATE\][\s\S]*?\[\/TASK_UPDATE\]\s*/g, '\n')
```

**问题**：
- 状态分散在多个 Map，没有统一的状态图
- 阶段转换逻辑隐式（`confirmGoal()` → `confirmPlan()` → ...），不显式可追踪
- 没有 checkpoint——Extension 重启后状态丢失

### 问题 3: ACP 事件流和 UI 渲染耦合

ACP 子进程 → `session/update` 回调 → `V2StreamHandler` → 直接拼 WebView postMessage 类型

```
ACP: agent_message_chunk  →  onText()  →  'stream-chunk'
ACP: agent_thought_chunk   →  onReasoning()  →  'thinking-chunk'
ACP: tool_call             →  onToolCall()  →  'tool-chunk'
```

WebView 收到就直接渲染了。**中间没有领域层**——消息没有经过正确的数据建模就被塞给了 UI。

## 目标架构

```
                     ┌──────────────────────────┐
                     │     Domain Model          │
                     │  (Task / Message / Phase) │ ← 唯一的真实源
                     └──────────┬───────────────┘
                                │
          ┌─────────────────────┼─────────────────────┐
          ▼                     ▼                     ▼
   ┌──────────────┐    ┌──────────────┐    ┌──────────────────┐
   │   Storage     │    │    Agent      │    │   UI State       │
   │  (SQLite)     │    │  ┌─────────┐  │    │  (collapsed,     │
   │               │    │  │ ACP     │  │    │   scroll, etc.)  │
   │               │    │  ├─────────┤  │    │                  │
   │               │    │  │ LangGraph│ │    │  ┌────────────┐  │
   │               │    │  └─────────┘  │    │  │ React/Van. │  │
   └──────────────┘    └──────────────┘    └──────────────────┘
```

**核心原则**：
1. **Domain Model 是唯一的 Task/Message 定义**——Store、Agent、UI 都基于这组类型
2. **UI 状态不混入 Domain Model**——`collapsed`、`streaming` 是渲染层的事
3. **Agent 状态不混入 Domain Model**——`sessionId`、`graphThreadId` 是 Agent 层的事
4. **`content` 永远是纯文本**——没有 JSON.parse
5. **LangGraph 模式从零构建，不继承 ACP 的任何协议包袱**——没有 `[TASK_UPDATE]`/`[TASK_DELEGATE]` 正则解析、没有 `<TODO_UPDATE>`/`<KNOWLEDGE_ENTRY>` JSON 内嵌、没有 `Map<string, boolean>` 手写状态集群。ACP 和 LangGraph 共享 Domain Model，但共享到类型定义为止，执行路径完全独立

## Domain Model 设计

```typescript
// ── 1. 核心：Task（持久化主体，统一任务和小助手）──
interface Task {
    id: string;
    title: string;
    type: 'task' | 'chat';     // 'chat' = 小助手（无阶段流），'task' = 任务模式

    // 需求 / 目标
    originalRequest?: string;
    goal: string;

    // 阶段流（与 LangGraph 节点一一映射）
    phase: Phase;
    status: TaskStatus;

    // 共识内容（与 Agent 类型无关）
    confirmedItems: string[];
    pendingItems: string[];
    planSteps: PlanStep[];
    riskItems?: string[];
    boundaryItems?: string[];

    // 元数据
    category?: TaskCategory;
    subType?: string;
    createdAt: number;
    workspace?: string;
    pinned?: boolean;
    group?: string;
    archived?: boolean;            // 侧边栏归档

    // 容器关系
    containerId?: string;          // 项目/分组树形组织

    // 跨模式共享
    hooks?: Partial<Record<Phase, string[]>>;  // 阶段钩子（AGENTS.md 定义）

    /* 移除字段说明：
     * - sessionId → 移到 Agent 层（符合原则 3）
     * - nodeMessageIds → 可从 type='phase_action' 消息时间线推导
     * - flowIteration → 已确认移除
     * - planVersion → 简化设计移除
     */
}

type Phase = 'goal' | 'plan' | 'execute' | 'self_verify' | 'review';
type TaskStatus = 'pending' | 'active' | 'in_review' | 'completed' | 'cancelled';

// ── 2. 消息（纯数据，无 UI 状态）──
interface Message {
    id: string;
    taskId: string;
    role: 'user' | 'agent' | 'tool';   // 保留 tool 角色
    type: 'text' | 'tool_call' | 'tool_result' | 'thinking' | 'phase_action';
    content: string;                    // 永远是纯文本，不需要 JSON.parse
    toolCall?: ToolCallInfo;            // type='tool_call' 时有效
    toolResult?: ToolResultInfo;        // type='tool_result' 时有效（LangGraph 的 ToolMessage 映射）
    phaseAction?: PhaseActionInfo;      // type='phase_action' 时有效
    timestamp: number;
}

interface ToolCallInfo {
    toolCallId: string;
    title: string;
    kind: string;          // 'bash' | 'read' | 'write' | 'edit' | ...
    status: 'running' | 'completed' | 'failed';
    // output 移至 ToolResultInfo，使用 toolCallId 关联
}

interface ToolResultInfo {
    toolCallId: string;    // 关联到对应的 ToolCallInfo
    output: string;
}

interface PhaseActionInfo {
    phase: Phase;
    status: 'pending' | 'confirmed' | 'rejected';
}

// ── 3. UI 状态（渲染层专属，不序列化）──
interface MessageUIState {
    streaming: boolean;
    collapsed: boolean;
    roundGroup: string | null;
}

interface AppUIState {
    viewMode: 'task' | 'assistant';
    activeTaskId: string | null;
    expandedRounds: Record<string, boolean>;   // roundGroup → 是否展开
    isGenerating: boolean;
    scrollLocked: boolean;
    // ... 其他纯 UI 状态
}

// ── 4. LangGraph StateGraph 状态（Agent 层专属）──
interface GraphState {
    messages: GraphMessage[];     // LangGraph 的 message 格式
    phase: Phase;
    status: TaskStatus;
    goal: string;
    confirmedItems: string[];
    planSteps: PlanStep[];
    taskId: string;
}
```

### 关键设计决策

**为什么小助手（`type: 'chat'`）和任务（`type: 'task'`）共享同一个 Task 模型？**
- 当前代码里小助手有独立的 `AssistantHandler`、`assistantPipeline.ts`、`StateManager`，导致小助手和任务模式的功能不一致——小助手没有折叠、没有 tool-use 循环
- 统一模型后，小助手 = 没有 phase 的任务，渲染层完全复用
- 折叠功能自然适用于小助手
- AI 走 LangGraph 时，小助手模式同样获得 tool-use 循环能力

**为什么 `toolCall` 和 `phaseAction` 不作为独立类型而作为 Message 的可选字段？**
- 消息是有序的（时间线），tool_call 和 thinking 是穿插在对话流中的
- 独立的 `ToolCallInfo` 导致需要额外维护顺序，而放在 Message 里自然有序
- 渲染时按 `type` 分发到不同组件即可

**为什么 `content` 必须是纯文本？**
- 当前代码约 **10 处** `try { JSON.parse(msg.content) }`（全局 grep 确认，仅限消息内容相关的，文件级 JSON.parse 不受影响）
- 每次解析都可能抛异常，失败后默默 return null 导致卡片不渲染
- `toolCall` 结构化字段彻底消灭这个模式

**LangGraph 的 GraphState 如何映射？**
- LangGraph 的 `StateGraph` 处理消息和阶段转换
- 但持久化仍然用 KCode 的 `Task` + `Message` 模型（更通用、不绑定框架）
- `GraphState` 只在 Agent 运行时存在，完成后同步回 TaskStore
- 这样即使未来换框架（或不换），数据层不受影响

## LangGraph.js 集成点

### 当前（ACP 模式）
```
                 TaskFlow.ts（手写状态机）
                      ↕
AgentService → AcpClient → ACP 子进程
                      ↕
               V2StreamHandler → postMessage
```

### 新增（LangGraph 模式）
```
                 LangGraph StateGraph（正式状态图）
                      ↕   checkpoint: better-sqlite3
AgentService → LangGraphProvider → LLM (直接 API)
                      ↕   streamEvents v2
               StreamAdapter → postMessage（复用现有格式）
```

**LangGraph 的节点映射**：
```typescript
const taskGraph = new StateGraph({
    channels: {
        phase: PhaseChannel(),            // 当前阶段
        taskId: TaskIdChannel(),           // 任务标识
        messages: MessagesChannel(),      // 消息历史
        goal: GoalChannel(),
        confirmedItems: ConfirmedChannel(),
        planSteps: PlanChannel(),
    }
})
.addNode('goal', goalNode)           // 格式化目标
.addNode('plan', planNode)           // 制定计划
.addNode('execute', executeNode)     // 执行
.addNode('self_verify', selfVerifyNode) // 自验
.addNode('review', reviewNode)       // 验收
.addConditionalEdges('__start__', (s) => s.phase || 'goal')
.addConditionalEdges('self_verify', (s) => s.passed ? 'review' : 'execute')
.compile({ checkpointer: new SqliteSaver(db) });
```

**共存**：ACP 模式和 LangGraph 模式都产生相同的 Domain Model（`Task` + `Message`），差异只在于：

| 维度 | ACP 模式 | LangGraph 模式 |
|------|---------|---------------|
| Agent 运行位置 | 子进程 | Extension 进程内 |
| 状态管理 | TaskFlow 手写 | StateGraph + Checkpointer |
| 消息产生 | ACP 流 → V2StreamHandler | streamEvents → StreamAdapter |
| 阶段转换 | [TASK_UPDATE] 协议解析 | Node 条件边 + interrupt |
| 恢复机制 | session/resume (ACP) | SQLite checkpoint |

**重要声明：LangGraph 模式下自定义协议块全部废弃**

在 ACP 模式下，AI 输出的文本流中嵌入了一系列自定义协议块供 TaskFlow.ts 正则解析：

| 协议 | ACP 模式 | LangGraph 模式 |
|------|---------|---------------|
| `[TASK_UPDATE]` 阶段迁移 | 正则解析 | **StateGraph 条件边自动处理** |
| `[TASK_DELEGATE]` 任务委派 | 正则解析 | **tool call**: `delegateTask(title, goal)` |
| `<TODO_UPDATE>` 待办 | 正则解析 JSON | **tool call**: `updateTodos(items, action)` |
| `<KNOWLEDGE_ENTRY>` 知识 | 正则解析 JSON | **tool call**: `saveKnowledge(entries)` |

**LangGraph 模式从零构建，不继承任何 `[TASK_*]`/`<*_UPDATE>`/`<KNOWLEDGE_*>` 协议**。所有结构化交互改为 tool call + StateGraph 状态更新。ACP 模式继续保留现有正则逻辑，不做迁移。

### Iteration 3: 渲染组件化 + 双管线统一（1-2 iterations）

**目标**：将 `createMsgElement()` 拆分为独立渲染器，并统一任务管线和小助手管线的渲染路径。

**当前问题**：`assistantPipeline.ts` 有自己的 `initAssistantPipeline()` 和独立的 `StateManager`，但渲染逻辑和 `renderManager.ts` 几乎一样——`_handleStreamChunk`、`_handleThinkingChunk`、`_handleToolChunk` 实现完全相同。

**方案**：一套渲染器 + 一个 StateManager，两种模式只是 view 切换：

```typescript
// 注册表模式（不依赖 React，任务和小助手共用）
const renderers: Record<string, MessageRenderer> = {
    'text': textRenderer,
    'thinking': thinkingRenderer,
    'tool_call': toolCallRenderer,
    'phase_action': phaseActionRenderer,   // 任务模式专用
    'round_summary': roundSummaryRenderer, // 两种模式共用
};

// 统一的消息处理
function handleStreamChunk(text: string, taskId?: string) {
    const msgs = [...stateManager.state.messages];
    // ... 同样的逻辑，不管是什么模式
    stateManager.patch({ messages: msgs });
}
```

**改动范围**：
- `msgRenderer.ts` — 拆为 `renderers/` 目录，渲染器不依赖 task/assistant 上下文
- `assistantPipeline.ts` — 移除独立渲染逻辑，改为复用统一的消息处理器和渲染器
- `basePipeline.ts` — `finalizeStream`、`closeRound` 等公共函数不区分模式
- `renderManager.ts` — 任务模式特有的阶段卡片逻辑保持不变，但消息渲染走统一渲染器

**结果**：
- 小助手模式自动获得折叠能力（因为走同样的消息处理器和渲染器）
- 删除 `assistantPipeline.ts` 中的重复代码（`_handleThinkingChunk`、`_handleToolChunk`、`_handleStreamChunk`）
- 新增消息类型只需加一个渲染器文件，自动同时服务于两种模式

**验证**：
- 所有消息渲染效果与重构前一致（两种模式分别验证）
- 渲染器可独立测试
- 新增消息类型只需加一个文件

### Iteration 4: 折叠功能重做（1 iteration）

**目标**：在新的 Domain Model + 统一渲染器上实现折叠，**同时适用于任务模式和小助手模式**。

当前小助手模式没有折叠（`assistantPipeline.ts` 中没有调用 `_collapseAllRounds`），统一后自然获得。

此时折叠是**自然结果**而非手术式修补：
- `collapsed` 在 `MessageUIState` 中，不污染数据
- `roundGroup` 是派生映射，不写死进 Message
- `round_summary` 是独立渲染器，不和 cardMeta 混淆
- 同一套折叠逻辑同时作用于 task 和 chat 类型

**实现**：
- round 识别：从 `messages` 按 user 消息边界划分
- 展开状态：`AppUIState.expandedRounds` Map 管理
- 折叠触发：`stream-done` 时自动对最新 round 应用折叠
- 交互：RoundSummary 渲染器 → 点击 toggle expandedRounds

**验证**：
- 折叠后中间消息隐藏，round_summary 显示
- 展开动画平滑
- 多 round 独立控制

---

### Iteration 5: LangGraph Provider（1-2 iterations）

**目标**：在 `AgentService` 中增加 `'langgraph'` 类型。

**改动**：
- 新增 `@langchain/langgraph`、`@langchain/langgraph-checkpoint-sqlite` 依赖（已有 `better-sqlite3`）
- 新增 `src/core/LocalAgentProvider.ts` — 实现 LangGraph StateGraph
- `StateGraph` 节点：`goal` → `plan` → `execute` → `self_verify` → `review`（无 demand 节点，demand 仅作为条件边起始判断，见上文"共存"章节的协议声明）
- `process.env` 中的 LLM API key 配置（与 ACP 配置分开）
- 流式事件 → 通过 `StreamAdapter` 映射为现有 WebView postMessage 格式（`stream-chunk`、`thinking-chunk`、`tool-chunk`、`stream-done`）
- checkpoint → `better-sqlite3`，保存 GraphState

**不改动**：WebView 端完全无感知，因为 postMessage 协议不变。

**对小助手模式的影响**：小助手模式目前通过 `AssistantHandler` → `AgentService` → ACP 子进程实现（`AssistantHandler.ts:349-383`），纯文本 prompt → 纯文本回答，**没有 tool-use 循环**。切换到 LangGraph 后：
- 获得工具调用能力（读文件、写文件、执行命令）——小助手不再只是"陪聊"
- 流式交互自然支持
- Session 持久化（SQLite checkpoint，掉线后可恢复）
- 折叠功能因此更有价值——小助手的回答过程中也会有中间工具调用，完成后自动折叠

**验证**：
- 选择 LangGraph agent type 后，对话正常
- Agent 能调工具（readFile/writeFile/runTerminal）
- 阶段按 StateGraph 自动流转
- Extension 重启后从 checkpoint 恢复

**目标**：前 5 步完成后，架构已经清洁。UI 框架换不换 React 是独立决策。

  - `textRenderer` → `<TextMessage>`

  - `roundSummaryRenderer` → `<RoundSummary>`
- 构建产物被 Extension 加载，与现有 esbuild 管线共存
- 逐步替换，逐个验证

**如果不换 React**，当前架构已经可维护：
- Domain Model 清洁
- 组件化渲染器
- 状态单一路径
- 折叠功能正常

## 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| 新 Domain Model 与现有代码不兼容 | Iteration 1-2 期间编译报错 | 类型别名桥接，新旧并存逐步迁移 |
| ChatMessage.type 13 种枚举涉及多处 | 遗漏导致运行时错误 | Iteration 2 逐个确认每个 type 的引用位置 |
| LangGraph Provider 与 ACP 的事件流格式差异 | WebView 渲染异常 | `StreamAdapter` 层做格式转换，保证 postMessage 一致 |
| 小助手和任务统一后功能回退 | 小助手或任务某些功能失效 | 两种模式分别验证，测试用例覆盖 |
| Iteration 3（渲染）和 5（LangGraph）并行开发 | 两边同时改 Message 类型，merge 冲突 | Iteration 2 结束时冻结 Message 类型定义，作为契约 |
| 插件系统未同步适配新模型 | 插件消息处理异常 | Iteration 2 包含插件适配步骤表，逐个确认 |
