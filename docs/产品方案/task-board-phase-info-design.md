# 任务看板阶段信息展示方案

## 一、概述

V3 任务看板（control panel）采用 **六阶段卡片** 布局，每个阶段卡片展示该阶段的**核心信息** + **行内输入**。主输入框仅在 init screen（State A）存在，用于新建任务。进入任务后（State B），主输入框删除，所有交互在各自阶段卡片内完成。

## 二、布局结构

```
.main-task-board
├── 任务标题行（任务名称）
├── #stage-cards
│   ├── demand     [▶ ✅ 需求提取           ✓3 待2]     ← header (始终可见)
│   │   ├── 已确认项列表                              ← body (可折叠)
│   │   ├── 待确认项列表 (可勾选确认)
│   │   └── ─── 📝 需求补充...            [发送]       ← 行内输入
│   ├── goal       [▶ 目标锚定             🎯 已确认]
│   ├── plan       [▶ 计划编排             2/4]
│   ├── execute    [▶ 代码执行             ⏱ 45s 📁3]
│   ├── verify     [▶ 自动化自验           3/5 ✅]
│   └── review     [▶ 最终签署             📁2 待签]
│
├── #chat-scroll   (只读聊天流，显示所有阶段交互记录)
```

## 三、阶段编号

六阶段自然编号：

| 编号 | 阶段 | 标识 | 内部步骤编号前缀 |
|------|------|------|----------------|
| 1 | 需求提取 | demand | — |
| 2 | 目标锚定 | goal | — |
| 3 | 计划编排 | plan | `3.1`, `3.2`, `3.3` ... |
| 4 | 代码执行 | execute | — |
| 5 | 自动化自验 | verify | — |
| 6 | 最终签署 | review | — |

编号前缀中的数字即阶段序号（`3` = 计划阶段），每个阶段内部只有**平铺列表**，无树形嵌套。

## 四、卡片结构（统一模板）

每个阶段的卡片统一由三部分组成：

```
┌────────────────────────────────────────────────────────────────┐
│ ▶ ✅ 阶段名称                            ✓3/5    ⏱ 1m23s     │  ← Header
├────────────────────────────────────────────────────────────────┤
│  阶段核心信息（聚焦重要数据，非所有数据）                        │  ← Body
│                                                                │
│ ────────────────────────────────────────────────────────────── │
│ 📝 快捷输入...                                    [发送]       │  ← 行内输入
└────────────────────────────────────────────────────────────────┘
```

### 4.1 Header

| 元素 | 说明 |
|------|------|
| `▶` | 折叠箭头（展开时旋转90°） |
| `✅` | 状态图标（success / running / pending） |
| `阶段名称` | 如"需求提取" |
| `✓3/5` 等 | **header badge** — 各阶段不同的进度摘要（见下表） |
| `⏱ 1m23s` | 持续时间（monospace） |

#### Header badge 各阶段定义

| 阶段 | badge 内容 | 含义 |
|------|-----------|------|
| demand | `✓3 待2` | 3 项已确认，2 项待确认 |
| goal | `🎯 已确认` 或 `⏳ 待确认` | 目标是否已锚定 |
| plan | `2/4` | 4 个步骤中完成 2 个 |
| execute | `⏱ 1m23s` 或 `📁3` | 执行耗时 / 文件改动数 |
| verify | `3/5 ✅` | 5 个测试通过 3 个 |
| review | `📁2 待签` | 2 个变更文件待签署 |

### 4.2 Body（核心信息展示）

| 阶段 | body 展示内容 | 用户操作 |
|------|-------------|---------|
| demand | 已确认项列表、待确认项列表（可勾选） | 勾选确认、填写需求补充 |
| goal | 目标文字、风险项、边界项 | 查看、输入调整 |
| plan | 步骤列表（平铺编号 3.1/3.2/...）+ 进度条 | 拖拽重排、确认/锁定 |
| execute | 文件变更列表、terminal 关键输出 | 干预指令、暂停/回滚 |
| verify | 测试列表（通过/失败）、迭代计数 | 查看详情 |
| review | 变更文件、验收 checklist | 勾选确认、评审意见 |

**原则**：只显示对用户来说重要的信息，不追求全部数据展示。

### 4.3 行内输入

每个阶段卡片底部嵌入一个小型输入框：

- 输入的消息进入**共同聊天流**，自动附带阶段标签 `[需求阶段]`、`[计划阶段]`
- AI 回复时自动感知当前阶段上下文
- 发送后卡片 body 显示简短"已发送"反馈
- 主输入框在进入任务后不显示

## 五、折叠策略

| 状态 | 默认行为 |
|------|---------|
| 当前 active 阶段 | **展开** |
| 已完成阶段 | 折叠（header badge 可见进度） |
| 未开始阶段 | 折叠 |
| 用户手动操作 | 可展开任意阶段查看历史信息 |

## 六、暂停功能

### 暂停按钮位置

放主 header，与阶段计数同行右侧：

```
KCode 任务面板  [● 任务攻坚中]  ✳ 需求开发  模型: Kilo
                                    ⏱️ 阶段: 3/6  ⏸ ▸
```

### 暂停时行为

- Rail 节点停止脉冲动画
- 所有卡片 body 只读（不可勾选、不可编辑）
- header 仍可折叠/展开
- 行内输入的消息**排队存储**，恢复后自动续发

## 七、空状态引导

全新任务（confirmedItems 和 pendingItems 均为空）：

| 场景 | body 展示 |
|------|----------|
| 全新任务（items 全空） | 居中引导 + 输入框："添加初始需求项以启动任务" |
| 已有需求（有 items） | 展示列表 + 底部小输入框"补充需求..." |

## 八、原始需求展示

用户的原始输入（在 init screen 输入的任务描述）需要保留并展示。这是任务发起的源头，也是需求提取阶段的核心上下文。

### 存储方案

在 Task 接口新增 `originalRequest` 字段，任务创建时从 init screen 输入写入，后续**不再变动**（即使任务重命名）。

```typescript
interface Task {
    originalRequest?: string;  // init screen 输入的原始需求，只写一次
}
```

### UI 位置

放在 **demand 卡片 body 顶部**，已确认/待确认列表之上，作为只读引用块：

```
┌─────────────────────────────────────────────┐
│ ▶ ✅ 需求提取                   ✓3 待2      │
├─────────────────────────────────────────────┤
│ 📋 原始需求                                  │
│ ┌─────────────────────────────────────────┐ │
│ │ 帮我实现一个用户登录功能，                  │ │
│ │ 支持邮箱和微信 OAuth 登录，                │ │
│ │ 以及密码重置流程                          │ │
│ └─────────────────────────────────────────┘ │
│                                             │
│ 已确认需求（3）：                             │
│  ✅ 实现邮箱登录                             │
│  ✅ 实现微信 OAuth 登录                      │
│  ✅ 密码重置流程                             │
│                                             │
│ 待确认需求（2）：                             │
│  ◻️ 登录失败次数限制                         │
│  ◻️ 短信验证码登录                           │
│                                             │
│ ── 📝 需求补充...              [发送]       │
└─────────────────────────────────────────────┘
```

### 特殊状态

| 场景 | 展示 |
|------|------|
| 有 `originalRequest` | 展示在引用块中 |
| `originalRequest` 为空 | 不展示该区域，demand 卡片直接从输入引导开始 |

## 九、交互数据流

```
用户在 demand 卡片输入 "补充 OAuth 需求"
  → postMessage({ type: 'sendMessage', text, phase: 'demand' })
  → 后端存入消息（带阶段标签）
  → 后端调用 AI（注入 demand 上下文）
  → AI 回复 → 更新 confirmedItems/pendingItems
  → 前端 updateTaskInfo → 重新渲染 demand 卡片
  → 聊天流追加消息记录
```
