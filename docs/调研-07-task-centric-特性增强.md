# 以任务为中心的差异化特性增强

> 调研日期：2026-05-23
> 状态：持续迭代中
> 核心判断：KCode 已建立任务中心基础，但在资产绑定、流程编排、约束治理等维度可进一步拉开与普通 Session 工具的差距。

---

## 概览：任务 vs 普通 Session 的差异

| 维度 | 普通 AI 工具（Session） | KCode 当前任务体系 | 差距 |
|------|------------------------|-------------------|------|
| 生命周期 | 在线/离线 2 状态 | 6 阶段状态机（D→G→P→E→V→R） | 小 |
| 资产 | 无绑定 | PTY（规划中）+ 设备连接 | 中 |
| 权限 | 无 | 无 | 大 |
| 编排 | 平铺指令 | 线性 5 阶段 | 中 |
| 沉淀 | 日志清零 | Wiki 导出 + Timeline | 中 |
| 关联 | 孤立 | 项目绑定 + FileChange | 小 |

---

## 一、任务专属生命周期体系

### 1.1 多阶段状态流转

| 状态 | 说明 | 实现状态 |
|------|------|---------|
| demand | 需求收集 | ✅ 已实现 |
| goal | 目标确认 | ✅ 已实现 |
| plan | 计划制定 | ✅ 已实现 |
| execute | 执行阶段 | ✅ 已实现 |
| self_verify | AI 自验 | ✅ 已实现 |
| review | 用户验收 | ✅ 已实现 |
| completed | 完成 | ✅ 已实现 |
| cancelled | 取消 | ✅ 已实现 |

**当前覆盖**：6 个活动阶段 + 2 个终态，附带完整状态迁移记录（`TimelineEntry`）。

**对比 Session**：普通 AI 工具只有"正在回答/回答完毕"两个隐含状态。

### 1.2 任务时效锁

| 功能 | 说明 | 实现状态 |
|------|------|---------|
| 有效期设定 | 任务创建时指定有效期 | ❌ 未实现 |
| 超时冻结 | 到期自动冻结会话，释放 PTY 资源 | ❌ 未实现 |
| 活跃度跟踪 | 记录最后活跃时间，超时预警 | ❌ 未实现（已有 `createdAt` 但无过期检查） |

**价值**：防止 PTY 僵尸进程堆积，尤其多任务并开时。

### 1.3 版本迭代快照

| 功能 | 说明 | 实现状态 |
|------|------|---------|
| 执行前快照 | 自动保存当前状态 → 执行 → 失败可回滚 | ❌ 未实现 |
| 版本管理 | 同一任务保留多次迭代版本 | ❌ 未实现 |
| 版本回溯 | 切换到历史版本查看/恢复 | ❌ 未实现 |

**已有基础**：`WikiExporter` 能做只读归档，`TimelineEntry` 记录事件序列，但无结构化版本管理。

**建议**：引入 `checkpoint save/restore`，执行前自动快照 task state + 文件变更 + PTY 输出，回滚时恢复到快照点。

## 二、任务绑定专属资产

### 2.1 上下文资产绑定

| 资产类型 | 绑定方式 | 实现状态 |
|---------|---------|---------|
| 终端会话 (PTY) | 每个 task 独立 node-pty 实例 | 🛠️ P26 规划中 |
| 设备连接 | `deviceClients: Map<string, IDeviceClient>` | ✅ 已实现（`handleDeviceConnect`/`Disconnect`） |
| 打开的文件 | 任务对话中 AI 读写的文件列表 | ✅ 已实现（`FileChange[]` 记录）|
| 环境变量 | 任务相关的 env 快照 | ❌ 未实现 |
| 配置参数 | 任务模板参数、初始化配置 | 🛠️ 部分已实现（`templates.ts inputFields`） |

**当前机制**：设备连接已绑定到 task（`deviceClients` Map keyed by taskId），PTY 正在规划。打开的文件在对话中引用但未显式留存为资产列表。

### 2.2 私有依赖缓存

| 功能 | 说明 | 实现状态 |
|------|------|---------|
| 依赖隔离 | 各任务 NPM/Python 缓存互不干扰 | ❌ 未实现 |
| 编译产物缓存 | 同任务重复执行复用编译产物 | ❌ 未实现 |
| 清理策略 | 任务删除时同步清理缓存 | ❌ 未实现 |

**值得注意**：PTY 共享同一工作目录已天然复用本地缓存，无需额外实现。仅当需要隔离编译环境（不同 Node 版本、不同 venv）时才有此需求。

### 2.3 标签分类归档

| 功能 | 说明 | 实现状态 |
|------|------|---------|
| 分类体系 | 4 大类 × 5 子类 | ✅ 已实现（`TaskCategory` + `subType`） |
| 类型标签 | `requirement_dev` / `code_review` / `problem_analysis` / `defect_analysis` / `log_analysis` | ✅ 已实现 |
| 分类筛选 | 侧边栏/我的任务面板按类型过滤 | ✅ 已实现 |
| 业务标签 | 用户自定义标签（如"部署"、"排错"） | ❌ 未实现 |

**当前覆盖**：预设分类体系完整，缺用户自定义标签。

## 四、任务流程编排能力

### 4.1 子任务嵌套拆分

| 功能 | 说明 | 实现状态 |
|------|------|---------|
| 委派拆分 | 对话中让 AI 拆出新任务 | ✅ 已实现（`TASK_DELEGATE`） |
| 父子层级 | 子任务关联父任务 ID，共享上下文 | ❌ 未实现（当前平级） |
| 子任务汇总 | 父任务查看子任务进度汇总 | ❌ 未实现 |
| 上下文共享 | 子任务继承父任务 PTY / 设备连接 | ❌ 未实现 |

**当前状态**：委派创建的是独立平级任务，无父子关联。可增强为显式的父子层级。

### 4.2 条件分支执行

| 功能 | 说明 | 实现状态 |
|------|------|---------|
| 退出码判定 | 根据 exit code 自动决定下一步 | ❌ 未实现 |
| 输出匹配 | 根据命令输出内容跳转到修复分支 | ❌ 未实现 |
| 重试策略 | 失败自动重试（最多 N 次） | ❌ 未实现 |
| 分支合并 | 修复后回到主流程继续 | ❌ 未实现 |

**当前状态**：5 阶段线性流转，无分支。适合引入规则引擎：AI 根据 exit code / 输出内容动态决策。

### 4.3 任务约束规则

| 规则类型 | 说明 | 实现状态 | 优先级 |
|---------|------|---------|--------|
| 目录白名单 | AI 只允许在指定目录写文件 | ❌ 未实现 | P0 |
| 命令黑名单 | 禁止 `rm -rf`、`sudo`、`:(){ :\|:& };:` | ❌ 未实现 | P0 |
| 执行超时 | 单条命令最大执行时间 | ❌ 未实现 | P0 |
| 资源上限 | 内存、CPU、磁盘配额 | ❌ 未实现 | P2 |

**判断**：这是当前 KCode 最大的安全缺口。**P0 优先级**，建议在 PTY 层实现命令拦截 + 超时控制。

## 七、综合评估：增强优先级

| 优先级 | 特性 | 维度 | 理由 |
|--------|------|------|------|
| **P0** | 任务约束规则（目录白名单 + 命令黑名单 + 超时） | 四.3 | 安全缺口，PTY 落地后必须配套 |
| **P1** | 任务时效锁 | 一.2 | 防止 PTY 僵尸进程 |
| **P1** | 变更 ↔ 终端操作溯源 | 六.2 | 低成本高感知，改动小 |
| **P2** | Checkpoint 版本快照 | 一.3 | 体验差异大但实现成本高 |
| **P2** | 子任务父子层级 | 四.1 | 依赖委派基础，可增量 |
| **P2** | 条件分支执行 | 四.2 | 编排能力核心，复杂度高 |
| **P3** | 执行质量评级 | 五.2 | 有价值但不紧急 |
| **P3** | 自定义标签 | 二.3 | 低成本，可做 |
| **P4** | 权限角色 / 观测数据 / 依赖缓存 | 三.1 / 六.3 / 二.2 | 多人协作或大规模场景才需要 |

## 附录：工作区初始化体检

打开代码工程时，AI 可自动扫描并为用户完成以下设置，无需手动配置：

| 类别 | 检测项 | AI 可做的 |
|------|--------|----------|
| **Git** | `user.name` / `user.email` / `.gitignore` / `.gitattributes` | 检测缺失自动提示；按语言生成标准 `.gitignore` |
| **Node.js** | `.nvmrc` / `.node-version` / `engines` in `package.json` | 检测版本要求 vs 当前版本，提示切换 |
| **Python** | `pyproject.toml` / `requirements.txt` / `Pipfile` | 检测活动 venv，推荐 `poetry shell` / `source venv/bin/activate` |
| **Go** | `go.mod` | 检测 go 版本，建议 `go mod tidy` |
| **包管理** | 锁文件缺失（`package-lock.json` / `poetry.lock`） | 自动 `npm install` / `poetry install` |
| **构建** | `tsconfig.json` / `vite.config.ts` / `Makefile` / `Cargo.toml` | 检测缺失，按技术栈建议默认配置 |
| **Lint/Format** | ESLint / Prettier / ruff / rustfmt / golangci-lint | 检测配置冲突，统一设置 |
| **Hook** | husky / pre-commit / commitlint / golangci-lint | 检测并初始化建议 |
| **VS Code** | `.vscode/settings.json` / `.vscode/extensions.json` | 推荐必需扩展，推荐格式化设置 |
| **Docker** | DevContainer / Dockerfile / docker-compose | 检测项目类型推荐 devcontainer |
| **环境变量** | `.env.example` / `.env` | 扫描 `process.env.*` 或 `os.getenv` 生成模板，检查 `.env` 是否泄露 |

**实现建议**：首条消息前做一次无声扫描（类似 `_autoDetectEnv`），汇总输出为"工作区体检卡片"注入到 AI prompt 第一轮上下文，用户无需任何手动配置。
