# KCode Task-Centric 功能定义

> 调研日期：2026-05-09
> 状态：📋 已定稿，待实现

---

## 核心理念

> **Chat 是为 Task 服务的，不是反过来。**

大多数 AI 编程工具是"聊天记录里附带任务信息"。KCode 应该是"任务生命周期里附带对话记录"。用户打开 KCode 看到的是任务的进展状态，而非一串聊天历史。

---

## 一、任务类型（保留双模式）

| 类型 | 用途 | 进度节点 | 适用场景 |
|------|------|----------|---------|
| **task** | 完整任务生命周期 | ✅ 显示 Goal → Plan → 步骤 → 验收 | 功能开发、Bug 修复、重构 |
| **chat** | 自由对话 | ❌ 不显示 | 问一个问题、闲聊、快速查询 |

选择逻辑：第一轮消息通过 `classifyIntent` 判断，用户也可以手动切换。

---

## 二、Goal 中心化（P0）

### 现状问题
Goal 只出现在首次确认卡片中，之后就消失了。后续被塞进 system prompt，用户不可见。

### 目标
**Goal 是任务的北极星，全程可见、可编辑。**

### 功能点

| # | 功能 | 说明 |
|---|------|------|
| G1 | **Goal 固定显示** | 对话区顶部始终显示当前任务目标，不随滚动消失（sticky header） |
| G2 | **随时修改 Goal** | 用户可点击编辑 Goal，修改后通知 AI 调整方向 |
| G3 | **Goal 与消息关联** | 每次 Goal 修改生成一条特殊消息，记录变更历史 |

### UI 示意

```
┌─────────────────────────────────────────────┐
│ 🎯 任务目标                                  │ ← sticky, 不滚动
│ 实现 CSV/JSON 导出功能，支持编码选择          │
│ [修改目标]                                   │
├─────────────────────────────────────────────┤
│ [对话区...]                                   │
│                                              │
```

## 三、Plan-before-Execute（P0）

### 现状问题
用户确认 Goal 后，AI 直接开始执行。用户不知道 AI 打算怎么做，中途也无法跟踪进度。

### 目标
**先计划，后执行。计划即路线图，贯穿整个任务生命周期。**

### 功能点

| # | 功能 | 说明 |
|---|------|------|
| P1 | **AI 提案 Plan** | 用户确认 Goal 后，AI 自动生成执行计划（步骤列表） |
| P2 | **Plan 结构化展示** | Plan 显示为有序列表，非纯文本。每步有状态：⬜ 待做 / 🔄 执行中 / ✅ 已完成 |
| P3 | **用户批准 Plan** | 只有用户点击"按计划执行"后，AI 才开始干活 |
| P4 | **可调整 Plan** | 用户可要求 AI 修改 Plan（增减步骤、调整顺序） |
| P5 | **Plan 持久化** | Plan 存入 TaskStore，不是内存临时变量。切换任务后恢复 |
| P6 | **Plan 实时更新** | AI 执行完一步，Plan 中对应项立即打勾，状态推送到前端 |

### UI 示意

```
Agent: ┌─────────────────────────────────┐
       │ 📋 执行计划                       │
       │                                  │
       │ ⬜ 1. 添加导出格式配置到 Settings  │
       │ ⬜ 2. 实现 exportToCSV()         │
       │ ⬜ 3. 实现 exportToJSON()        │
       │ ⬜ 4. 添加菜单入口               │
       │ ⬜ 5. 实现下载逻辑               │
       │                                  │
       │ [✅ 按计划执行]  [↩ 调整计划]      │
       └─────────────────────────────────┘
```

## 四、进度节点面板（P0）

### 现状问题
任务进展没有可视化。用户不知道 AI 执行到哪一步了，也无法快速跳转到历史关键节点。

### 目标
**每个 Task 有一个可视化时间线，节点即导航。**

### 功能点

| # | 功能 | 说明 |
|---|------|------|
| N1 | **节点面板** | 对话区左侧新增一个面板，显示当前任务的进度时间线 |
| N2 | **节点类型** | 每个节点代表一个阶段：`🎯 Goal` → `📋 Plan` → `⚡ Step N` → `✅ 验收` |
| N3 | **状态可视化** | 每个节点有明确状态：⬜ 待处理 / ▶ 进行中 / ✅ 完成 / ❌ 失败 |
| N4 | **点击跳转** | 点击已完成节点，对话区自动滚动到对应消息位置并高亮 |
| N5 | **当前节点高亮** | 正在执行的节点有脉冲动画，一目了然 |
| N6 | **Chat 类型隐藏** | Chat 类型的任务不显示节点面板 |
| N7 | **面板可折叠** | 用户可收起节点面板，给对话区更多空间 |

### UI 示意

```
┌────────────┬──────────────────────────┬──────────┐
│ 📋 进度     │ 对话区                   │ Preview  │
│             │                          │          │
│ 🎯 Goal ✅  │ 🎯 任务目标              │          │
│  确认目标    │ 实现导出功能...          │          │
│             │                          │          │
│ 📋 Plan ✅  │ ── AI 提出计划 ──        │          │
│  批准计划    │ 📋 执行计划...            │          │
│             │ [✅ 按计划执行]           │          │
│ ⚡ Step1 ✅  │                          │          │
│  配置 Settings│ ── AI 执行中 ──          │          │
│             │ 🔧 正在修改 settings.ts... │          │
│ ▶ Step2    │                          │          │
│  实现CSV    │                          │          │
│             │                          │          │
│ ⬜ Step3    │                          │          │
│  实现JSON   │                          │          │
│             │                          │          │
│ ⬜ 验收     │                          │          │
│             │                          │          │
└────────────┴──────────────────────────┴──────────┘
```

## 五、结构化验收（P0）

### 现状问题
当前验收是笼统的通过/驳回。用户需要自己对比 goal 来判定是否完成。

### 目标
**验收逐条核对，不遗漏、不模糊。**

### 功能点

| # | 功能 | 说明 |
|---|------|------|
| R1 | **AI 标记完成** | 保留 `[TASK_STATUS: completed]` 机制，触发验收阶段 |
| R2 | **验收卡片结构化** | 显示：完成摘要 + 变更文件列表 + 逐条验收标准 |
| R3 | **验收标准自动生成** | AI 根据 Goal + Plan 自动提炼验收标准（/ 用户可手动补充） |
| R4 | **逐条核对** | 每条验收标准独立开关：✅ 通过 / ❌ 不通过。不是笼统通过/驳回 |
| R5 | **驳回后精准迭代** | 仅未通过的标准重新进入执行，已通过的保留 |
| R6 | **变更文件关联** | 每条验收标准关联涉及的文件变更，点击跳转 diff |

### UI 示意

```
┌──────────────────────────────────────────────┐
│  ✅ AI 已完成任务                              │
│                                                │
│  📝 变更文件:                                   │
│    📄 src/export/csv.ts   (新建)               │
│    📄 src/export/json.ts  (新建)               │
│                                                │
│  ✅ 验收标准:                                   │
│    ☑ CSV 格式导出，字段完整                    │
│    ☑ JSON 格式导出，缩进可配置                 │
│    ☑ 编码支持 UTF-8 / GBK                      │
│    ☐ 大文件 (100MB+) 分块处理                  │
│                                                │
│  [✅ 通过验收]  [↩ 驳回未通过项]                │
│                                                │
│  ── 逐条点击可查看对应 diff ──                  │
└──────────────────────────────────────────────────┘
```

## 六、完整任务生命周期（全景）

把以上所有串起来，一个 task 类型任务的完整生命周期：

```
                       用户输入需求
                            │
                            ▼
                    ┌────────────────┐
                    │  AI 格式化 Goal │ ← 节点: Goal (pending→active→completed)
                    └───────┬────────┘
                            │ 用户确认
                            ▼
                    ┌────────────────┐
                    │  AI 提案 Plan   │ ← 节点: Plan (pending→active→completed)
                    └───────┬────────┘
                            │ 用户批准
                            ▼
                    ┌────────────────┐
                    │  执行 Step 1    │ ← 节点: Step 1 (pending→active→completed)
                    ├────────────────┤
                    │  执行 Step 2    │ ← 节点: Step 2 (pending→active→completed)
                    ├────────────────┤
                    │  执行 Step N    │ ← 节点: Step N (pending→active→completed)
                    └───────┬────────┘
                            │ AI 标记完成
                            ▼
                    ┌────────────────┐
                    │  结构化验收     │ ← 节点: Review (active)
                    └───────┬────────┘
                            │
                ┌───────────┴───────────┐
                ▼                       ▼
          ✅ 全部通过              ❌ 部分驳回
                │                       │
                ▼                       ▼
        ┌────────────┐         ┌────────────────┐
        │ completed   │         │ active (重做)   │
        └────────────┘         └────────────────┘
                                    │ 仅重做未通过项
                                    ▼
                             回到对应 Step 节点
```

## 七、与普通 AI 编程工具的核心差异

| 维度 | 普通 AI 工具（Chat-Centric） | KCode（Task-Centric） |
|------|---------------------------|---------------------|
| **入口** | 点击就聊 | 先选/建 Task，再对话 |
| **Goal** | 隐含在第一句话里 | **始终可见的 UI 组件** |
| **计划** | AI 自由发挥 | **必须先有 Plan，用户批准** |
| **进度** | 无，只有滚动消息 | **左侧节点面板，一目了然** |
| **验收** | 用户心理判断 | **逐条核对，结构化验收** |
| **导航** | 滚动找历史 | **点击节点跳转关键位置** |
| **Chat 模式** | 唯一模式 | 纯问答时可用，但不混淆 |

## 八、迭代路线

### Step 1 — 基础链路闭环（P0）
- Goal 固定显示（对话区顶部 sticky）
- Plan-before-Execute 流程打通
- Plan 持久化 + 状态更新
- 节点面板 V1（纯展示，点击跳转 V2）
- 结构化验收（逐条核对）

### Step 2 — 交互完善（P1）
- 节点面板点击跳转对话位置
- 编辑 Goal / 调整 Plan
- 驳回后精准迭代（仅重做未通过项）
- 面板折叠/展开

### Step 3 — 体验打磨（P2）
- 动画、高亮、过渡
- 面板宽度记忆
- 键盘快捷键（Tab 切换面板）
- 暗色主题适配
